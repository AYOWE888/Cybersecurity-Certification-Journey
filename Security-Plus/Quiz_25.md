CompTIA Security+ SY0-701 Module 3.3

**Subject:** Data Types, Classifications, and Protection Methods

**Source Material:** Professor Messer SY0-701 Training Series (Module 3.3)

---

### Key Points Studied & Tested

* **Data Classifications:** Distinguishing between Public, Confidential, Proprietary, PII (Personally Identifiable Information), and PHI (Protected Health Information).
* **States of Data:** Understanding security requirements for **Data at Rest** (Storage), **Data in Transit** (Network), and **Data in Use** (RAM/CPU).
* **Data Protection Techniques:** Implementation of Encryption (Confidentiality), Hashing (Integrity), Obfuscation (Logic hiding), Masking (Display protection), and Tokenization (Substitution).
* **Geographic Controls:** The impact of **Data Sovereignty** (Jurisdictional laws like GDPR) and the use of **Geo-fencing** to restrict access based on location.

---

### High-Impact Question Analysis

| # | Question Focus | Key Takeaway |
| --- | --- | --- |
| **5** | **Data in Use Vulnerabilities** | Data in RAM is typically unencrypted so the CPU can process it; this makes it the primary target for RAM scrapers. |
| **4** | **Tokenization Mechanism** | Unlike encryption, tokenization replaces sensitive data with a random substitute that has no mathematical link to the original. |
| **6** | **Data Sovereignty** | Data is legally bound by the laws of the country where it is physically stored, regardless of the owner's origin. |
| **23** | **Cryptographic Confusion** | Confusion ensures the relationship between the encryption key and the ciphertext is as complex as possible to prevent reverse engineering. |
| **9** | **Hashing vs. Encryption** | Hashing is a one-way function used for integrity; encryption is a two-way function used for confidentiality. |
| **12** | **Segmentation Strategy** | Dividing databases into smaller pieces prevents an attacker from gaining access to the entire data set through a single breach. |
| **2** | **Obfuscation Utility** | Obfuscation protects intellectual property by making code logic unreadable to humans while maintaining functional execution. |
| **49** | **Payment Security** | One-time use tokens in mobile payments prevent replay attacks, as the captured token cannot be reused for a second transaction. |
